## 1. プロダクトコンセプト

### 一文サマリー

**避難所に着いた人が、スマホを開くだけで開設チームの一員になれる。**

QRコードを読むと同じセッションに入り、自動で役割候補が提示される。各自が「今やること」を1ステップずつ案内され、やったことと困りごとは1タップで仲間に共有される。

### 製品コンセプト文(差別化を含む)

> マニュアルを"見る"から、みんなで"動く"へ。
> 役割分担と進捗共有で、知識ゼロでも避難所が立ち上がる。
> 事前防災の専門家が監修する、住民協働型の避難所運営プラットフォーム。

### 設計思想(3原則)

1. **1画面に1ステップだけ**(認知負荷を最小に)
2. **「次にやることが指で押せる」**(読ませず、行動させる)
3. **判断を3択以下に**(完了 / 困った / スキップ)

### 何をしないか

- 個別の災害判断・医療判断はアプリが行わない
- マニュアルの完全再現は目指さない(行動に絞る)
- 自治体システムの代替はしない(連携の前段に徹する)

## 2. ユーザー像と前提

### 主要ユーザー3層

| 層 | 想定 | 最重要度 |
|---|---|---|
| 初動メンバー | 最初に避難所に到着した近隣住民数名。スマホは使えるが防災知識はない | ◎ |
| 避難所リーダー / 自治体担当者 | 後から合流する経験者。全体の進捗を俯瞰 | ○ |
| 本部・自治体側 | 複数避難所をダッシュボードで把握 | △(MVPでは想定外) |

### 利用シーン

- **発災直後の避難所開設**(最初の数時間)
- **平時の訓練・研修での利用**

### 技術前提

- オンライン前提(クラウド同期)
- ただし発災直後の通信不安定を想定し、**楽観的UI(オフラインでもタップは進められ、後で同期)** を実装

### 最も重視するユーザー体験(優先順位)

1. 防災知識ゼロでも迷わず行動できる
2. 役割間の情報共有・進捗可視化
3. 自治体や本部との連携
4. 現場での課題発見・記録

## 3. コア機能の全体像

優先順位の高い4つを軸とする。

### (A) セッション参加とロール割り当て

- 代表者が避難所セッションを作成、または自治体が事前発行したQRを掲示
- 後続者はQR/コードで参加
- 参加時に3〜4の簡単な質問で役割を推薦(体力・医療経験・子どもとの相性など)
- 本人が選択。途中参加・役割変更・かけもち可

### (B) 役割別ステップガイド

- 各役割に「いま、次にやること」を1画面1ステップだけ表示
- 各ステップに「完了/スキップ/困った」の3ボタン
- 写真や図解で知識ゼロでも判断できるように

### (C) 進捗の見える化

- 全役割の進捗を1画面で俯瞰できるボード
- 誰がどの役割で何ステップ完了したか
- 未着手のミッションも一目で把握

### (D) 課題・発見の1タップ共有

- 「物資が足りない」「要配慮者がいる」など気づきを投稿
- 定型タグ+短文+写真
- 役割を越えて共有、(将来)本部にも自動連携

## 5. マイミッション画面の詳細設計

### 画面要素

1. **ヘッダー**:今の役割と仲間の人数(タップで仲間の進捗確認)
2. **フェーズ + ステップカウンター + プログレスバー**:残り何個でフェーズが終わるかを示す
3. **いまやること(黄色バッジ + 大見出し)**:1画面1ステップ、10〜15文字の動詞で終わるタイトル
4. **手順(3つまで)**:1行20文字以内
5. **写真で見る(任意)**:設置完了形の写真を1枚、タップで開く
6. **「できた」ボタン**:緑色、52px高さ。画面の主役
7. **「困った」ボタン**:常設。物がない/わからない/自分には難しい、を選択
8. **「スキップ」ボタン**:状況的に不要な場合
9. **下部ナビ**:マイ / 全体 / 共有(3タブのみ、共有に新着ドット)

### 重要な設計上の工夫

**ステップの粒度**
1ステップ = 1〜5分で終わる単位。大きな粒度は禁止。

**ステップ順序の柔軟性**
役割内のステップ一覧をスワイプで見られる。並列ステップは並列表示。

**「困った」が起点の学習ループ**
押された頻度・場所がダッシュボードに集計される。手順書改善のデータになる。

**「できた」を押すだけで報告完了**
誰が・何時に・どのステップを完了したかが自動でログ化される。

**オフライン時の挙動**
「できた」を押した瞬間に通信が落ちていてもローカル保存、復旧時に同期。

## 6. 役割設計とフェーズ別運用

### 基本役割(内閣府避難所運営ガイドラインを下敷きに)

- 総務班(受付・名簿)
- 情報班(本部連絡・情報発信)
- 施設班(スペース割り・安全確認)
- 食料物資班
- 救護衛生班(トイレ・体調不良者対応)
- 要配慮者支援班

### フェーズ別に役割を段階的に出現させる

| フェーズ | 時期 | 立ち上げる役割 |
|---|---|---|
| 0:初動 | 発災〜30分 | 施設・総務 |
| 1:開設初期 | 〜2時間 | + 情報班 |
| 2:応急運営 | 〜数時間 | + 食料物資・救護衛生 |
| 3:安定運営 | 1日〜 | + 要配慮者支援 |

### 役割推薦の3問質問(MVP案)

1. 体を動かす作業はできますか?(できる/少しなら/難しい)
2. 医療や福祉の経験はありますか?(ある/少しある/ない)
3. 人前で話すのは得意ですか?(得意/普通/苦手)

→ 結果から役割を1〜2つ推薦、最後は本人選択。

## 7. 役割別ステップコンテンツ(初動フェーズ)

### 共通ルール(編集ガイドライン)

- タイトル:動詞で終わる10〜15文字
- 所要時間目安:1〜5分(超える場合は分割)
- 手順:最大3行、各20文字以内
- 完了条件:「何ができていれば次に進めるか」を1行で
- 困ったときの選択肢:3つまで定型化
- 依存関係:前提となる他ステップがあれば明記

### 役割1:総務班(受付・名簿)

**ミッション**:避難者が安心して登録でき、誰がいるか把握できる状態をつくる。

**フェーズ0(到着〜15分)**
1. 受付スペースを決める
2. 避難者名簿を準備する

**フェーズ1(15分〜1時間)**
3. 最初の避難者を受け付ける
4. 要配慮者を救護班に引き継ぐ

**フェーズ2(1〜2時間)**
5. 受付列の流れを整える
6. 受付シフトを組む

### 役割2:施設班(スペース・安全)

**ミッション**:建物が安全に使えるか確認し、避難者が休むスペースを作る。

**フェーズ0(到着〜15分)**
1. 建物の被害を目視点検する
2. 使えない場所を立入禁止にする
3. 電気・水道・ガスを確認する

**フェーズ1(15分〜1時間)**
4. 居住スペースを区画する
5. 要配慮者ゾーンを別に確保する

**フェーズ2(1〜2時間)**
6. トイレの場所を決めて掲示する
7. 喫煙場所・火気禁止の表示を出す

### 役割3:情報班(本部連絡・情報発信)

**ミッション**:自治体本部とつながり、避難所の中で正しい情報を伝える。

**フェーズ0(到着〜15分)**
1. 本部への第一報を入れる
2. 避難所内の掲示板を作る

**フェーズ1(15分〜1時間)**
3. 避難者数を本部に報告する
4. 最新情報を掲示する

**フェーズ2(1〜2時間)**
5. 館内アナウンスを行う
6. 報道・問い合わせ対応の窓口になる

### 役割4〜6(MVP対象外、将来実装)

- 食料物資班:備蓄倉庫を開ける、物資をカテゴリ別に並べる、在庫数を記録、配布ルールを決める、配布列を作る、不足物資を本部に要請
- 救護衛生班:救護スペースを確保、トイレの状態確認、要配慮者リスト受領、持病・服薬の聞き取り、手洗い場所整備、隔離スペース準備
- 要配慮者支援班:要配慮者ゾーンへの常駐、乳幼児連れの情報共有、外国語話者対応、手話・筆談・絵カードの準備、見守りシフト

## 9. 競合分析と差別化戦略

### 競合の位置づけ

| プロダクト | 立ち位置 | 強み | 弱み |
|---|---|---|---|
| **N-HOPS(能美防災)** | 自治体向けPWA。アクションカードのデジタル化 | 老舗の信頼、PWA、自治体導入実績 | 自治体契約モデル、協働機能弱い、コンテンツ品質が自治体任せ |
| バカン | 避難所マネジメント、混雑可視化 | 視覚的わかりやすさ | 開設・運営支援は対象外 |
| 全国避難所ガイド | 住民向け検索アプリ | 認知度 | 運営側ではない |
| 紙のアクションカード | 多数の自治体が採用 | 安価、シンプル | 更新性が弱い、印刷し直し、共有不能 |

### N-HOPSの空白地点(差別化の機会)

1. 自治体が契約主体 → 住民・自主防災組織が自発的に使えない
2. 名簿管理機能なし
3. 「役割分担」「グループ参加」「進捗共有」が前面に出ていない
4. コンテンツ作成は自治体任せ or 有償代行 → 質が自治体に依存
5. 「困った」起点の改善ループが見えない
6. コミュニティ性・複数人協働の設計が薄い
7. 平時の防災教育・訓練の体験設計が弱い
8. 要配慮者・外国人・男女共同参画の視点が前面に出ていない

### 差別化戦略の組み合わせ

```
主軸:  協働運営 (役割分担・進捗共有・複数人参加)
副軸1:専門家監修コンテンツ
副軸2:平時の体験(訓練と本番が同じ画面)
将来軸:学習ループ(困った起点で全国の手順が育つ)
```

### 一文での差別化メッセージ

> N-HOPSは「マニュアルをデジタル化して、一人が見るためのガイド」。
> 私たちは「複数人が集まって、専門家監修の手順を一緒にこなし、平時から本番まで使える協働プラットフォーム」。
> ガイドの閲覧ではなく、**チームでの実行**を支援する。

### 5つの具体的差別化点

| 差別化点 | N-HOPS | 私たち |
|---|---|---|
| 同時参加 | 個別閲覧 | QRで複数人セッション |
| 役割設計 | ガイド単位 | 動的役割推薦・途中参加 |
| 進捗可視化 | 自分の進捗 | 全役割のリアルタイムボード |
| コンテンツ品質 | 自治体任せ | 専門家監修ライブラリ |
| 改善ループ | 自治体内編集 | 困った起点の横断学習 |

### 競合の強みへの対応(リスク管理)

- 老舗のブランド差 → 専門家としての個人ブランド+大学・自治体との共同研究で補強
- 機能差は模倣されやすい → 模倣困難な3資産に重点(専門家ネットワーク、横断学習データ、平時コミュニティ運営)
- 営業力差 → 自治体契約に正面衝突せず、住民起点モデルを併用

## 10. 既存ツールが解決していない5領域と新機能アイデア

### 領域1:公平性と紛争予防

**課題**:食料配分のトラブル、奪い合い、盗難、ルール作成の遅延
**機能アイデア**
- 公平配布アルゴリズム(構成から優先順位を自動算出、透明性を担保)
- 見える名簿×物資マッチング(必要なものを自動集計、本部発注精度向上)
- ルールの即時テンプレート(配布・使用・夜間ルールを1タップ生成)

### 領域2:多様性ニーズへの即応

**課題**:外国人対応の現場任せ、障害者支援の放置、性的マイノリティへの配慮不足、福祉避難所移送の遅延
**機能アイデア**
- やさしい日本語・ピクトグラム選択型受付
- 個別ケアプラン自動生成
- 福祉避難所との連動キュー
- ピクトグラム掲示ジェネレーター

### 領域3:心理的ケアと空気のマネジメント

**課題**:ストレス、騒音問題、感染症ストレス、運営者バーンアウト
**機能アイデア**
- 心理的安全のチェックイン(3タップで気分記録)
- 騒音・温度の見える化(個人を責めず環境のせいにできる)
- 運営者のバーンアウト警告(連続稼働時間と困った頻度から)
- 子ども・乳幼児ゾーン自動立ち上げ

### 領域4:引き継ぎと連続性

**課題**:シフト交代での情報断絶、職員参集との引き継ぎ、報道対応の不統一、被災者の追跡不能、帰宅困難者の急増
**機能アイデア**
- 3分引き継ぎカード(口述録音+定型項目)
- 共通回答スクリプト(本部承認済み)
- 避難者ライフサイクル管理(登録〜仮設住宅まで)
- 帰宅困難者の自動ルーティング

### 領域5:「中の現実」の即時把握

**課題**:人数のリアルタイム把握困難、トイレ汚れの放置、物資在庫の不正確、体調不良の発見遅延
**機能アイデア**
- "今ここ"カウンター(QRでの入退所記録)
- トイレ・水場のQRフィードバック
- 物資の見たまま在庫
- 体調変化の集合検知

### 飛び道具系アイデア

- **外部専門家の遠隔参加**:看護師・心理士・通訳などが自宅から複数避難所を支援
- **次の避難所の設計学習ループ**:本番運営ログから事前設計に還元
- **コミュニティ事前スキル登録**:発災時に「ここに看護師がいる」が分かる

## 11. 4方向の情報フロー

このアプリは単なる手順支援ではなく、避難所が持つ4つの情報フロー全てを円滑にする情報基盤を目指す。

### 流れ1:避難者 → 避難所(受付)

**目指す状態**:30秒で1人の登録完了、要配慮者の自動抽出
**機能**
- スマホ自己受付(避難者自身が入力)
- やさしい質問型受付(1問ずつ、選択式)
- 自動分類と即時通知(要配慮者は救護班、ペット連れは施設班に自動通知)
- 連絡先と安否情報の連動(将来)
- 紙との併用(通信不通時・高齢者向け)

### 流れ2:避難者の中で(協働運営)

これまでに議論したコア機能群。
- QRで同一セッション参加
- 役割推薦と動的割り当て
- マイミッション画面
- 進捗ボード
- 「困った」「課題発見」の1タップ共有
- 引き継ぎカード

### 流れ3:避難所 → 行政(報告)

**目指す状態**:本部が全避難所の現況を1画面で把握、報告作業ゼロに近づく
**機能**
- 自動レポート(現場入力データから本部に自動送信)
- 30分定時報告 + 異常時即時通報
- 写真つき状況報告
- 物資要請の構造化(本部側で自動集約)
- 本部側ダッシュボード(地図上で俯瞰)

### 流れ4:行政 → 避難所(配信)

**目指す状態**:本部からの情報が避難者全員に正確に届く
**機能**
- 多重チャンネル配信(アプリ通知・館内掲示自動生成・情報班通知)
- 「確認しました」の追跡
- 多言語自動翻訳(日→英・中・韓・ベトナム・ポルトガル・やさしい日本語)
- 個別配信(乳幼児連れ・ペット連れ・要配慮者など属性別)
- ライフライン情報の集約(自分の地域の復旧見込み)
- 罹災証明・公的支援の段階的案内

### スコープ拡張時の重大論点

- 行政システムとの連携の壁(自治体ごとに個別開発が必要)
- 自治体の調達ハードル(行政連携機能は正式採用が必要)
- 災害時の通信障害(クラウド前提では一番大事な瞬間に動かない)

→ 対策:**段階的機能解放**(住民だけで使う場合は1と2のみ、自治体採用で3と4が解放)


## 13. 検証計画と成功指標

### 検証方法

1. 協力自治体を1つ確保(専門家ネットワーク経由)
2. 訓練を3回実施(同じ地域・同じ施設、住民メンバーを変える)
3. 観察と計測

### 計測項目

- 開設行動が始まるまでの時間
- どのステップで詰まったか
- 「困った」が押された回数と内容
- 全ステップ完了までの時間
- 参加者の事後アンケート

### 成功指標(MVPの合格ライン、仮)

- **行動着手率**:防災知識ゼロの参加者の8割以上が、役割を選んで最初のステップに着手できた
- **初動立ち上げ時間**:受付・建物点検・本部連絡の3つが、訓練開始から30分以内に立ち上がった
- **継続意欲**:参加者の8割が「実際の災害でも使いたい」と回答
- **詰まりポイント特定**:「困った」が集中したステップが3〜5個特定された(改善材料)

### 検証で見たい質的データ

- 役割選択で迷う人がどれくらいいたか
- 「できた」を押すタイミングの個人差
- 他の役割の進捗を見る頻度
- 訓練後に話したくなる出来事(やりがい・気づき)

## 16. 倫理的・法的留意点

### 個人情報

- 避難者名簿の取り扱い:自治体個人情報保護条例との整合
- 要配慮者情報は機微情報として扱う
- 写真共有時の肖像権・プライバシー
- 保存期間と自動削除のルール

### 法的位置づけ

- 災害対策基本法上の避難所運営者の責任範囲とアプリの位置づけ
- アプリの指示によって生じた事故の責任(免責条項の設計)
- 自治体との利用契約・データ所有権

### 倫理原則(将来機能用)

- 気分チェックイン・心理データ:個人特定不可、第三者非開示、本人の意思で削除可
- 騒音マイク:音量のみで音声内容は記録しない、を設計段階で担保
- 外部専門家の遠隔参加:アクセス権限の細かい設計、避難者の同意取得
- 事前スキル登録:任意性の徹底、撤回権の保証
